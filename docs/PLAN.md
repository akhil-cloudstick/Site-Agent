# Plan: SiteAgent v1 — Slice 1 (the walking skeleton)
_Locked via grill — by Claude + Dineshraj. Architecture is fixed by `Architecture.md` (Codex-approved) and is NOT re-litigated here. This plan only sequences the FIRST vertical slice and its execution choices._

## Goal
Stand up the thinnest end-to-end vertical slice of the SiteAgent "Brain" that proves the **core editing loop works safely**: one Tenant logs in → types a content edit in chat → a Gemini-backed agent turns it into a structured intent → the Tool broker applies it through the **single audited Local-API adapter** (`overrideAccess:false`, minimal-role service principal) → the shared `beforeChange` hook stamps it into the Tenant's active ChangeSet as a draft → the change renders back in an **in-Brain preview** pane. The slice deliberately stops at preview (no publish, no Cloudflare, no media). Its real payoff is proving the **load-bearing isolation + ChangeSet controls** end-to-end on `localhost`, not shipping a public site. Executed task-by-task against `../.serve/ProjectPlan.html` in dependency order (Option B, vertical-slice-first).

## Approach
Build order (dependency-correct; each step is a tracker task or a marked slice-1 substitute):

**Step 0 — tracker truth (do this first, Codex R2 #1).** Before any code, update `../.serve/ProjectPlan.html`: add the slice-1 substitute task `m8-inbrain-preview` and **override the dependency** so `m13-workspace` depends on `m8-inbrain-preview` (not `m8-ssr-worker`) for slice 1, and bump `CONTENT.contentVersion` + `generatedFrom` per the file's own bump rule. Until this edit lands, the tracker's `DEPS` graph and this plan disagree — so it is the literal first action, not an assumed state.

**Foundation**
1. `m1-scaffold` — one Next.js + Payload v3 codebase, one process, one Postgres connection to **local installed Postgres** via `DATABASE_URL` in `.env`. Package manager **pnpm**, Node 20+. Include a **reset-and-reseed** script (drop/recreate dev DB, re-run migrations, re-seed) for fast iteration.
2. `m1-config-seam` + `m11-secrets` — server-side-only secret seam (`DATABASE_URL`, `GEMINI_API_KEY`); nothing reaches the client bundle. A typed config module is the only reader of `process.env`.

**Data model (hard constraints, not convention)**
3. `m2-payload-collections` — minimal content model: a `pages` collection with a tiny editable set (e.g. a hero `heading`/`subheading` + a couple of text fields) sufficient to demo "change the hero." Every tenant-scoped row carries `NOT NULL tenant` + `NOT NULL changeSetId`; drafts/versions on.
4. `m2-tenants` — tenants collection extended with Site attrs (slug, status). (githubRepo/deployTargets columns exist but unused in slice 1.)
5. `m2-changesets` — the join key + state machine. Implement the **FULL status enum** from `DB-Architecture.md` now (`active, previewing, publishing, published, aborted, deployed_pending_publish, failed_deploy_reverted, rolled_back_from_deployed_pending_publish`) and the **full blocking partial index** `UNIQUE(siteId) WHERE status IN ('active','previewing','publishing','deployed_pending_publish')` — even though slice 1 only ever transitions `active → published|aborted`. Building the correct constraint now is drift-proof; a slice-1-only subset would be wrong (`published`/`aborted` are terminal NON-blocking). The blocking-set is **one shared constant** imported by both the migration and the guard (step 6), asserted equal by a test. _(Codex R1 #1.)_
6. `m2-changeset-guard` — guarded transition code rejecting illegal moves; blocking-set MUST equal the partial-index set (one shared constant).
7. `m2-fk-constraints` + `m2-migrations` — `NOT NULL tenant` FK on every tenant-scoped table (schema-layer isolation); single migration history (Payload-owned DDL + control-plane constraints as custom SQL inside Payload migrations). Never hand-edit the DB.

**Payload + the ChangeSet hook (Contract B)**
8. `m4-multitenant` — multi-tenant plugin (row-level scoping) on content collections.
9. `m4-drafts` — drafts/versions on content collections. **Disable autosave in slice 1** (autosave/admin-version write paths can invoke hooks differently and create version rows without the ChangeSet stamp). A test asserts **every** write path — create, update, and version rows — carries `NOT NULL tenant` + `changeSetId`. _(Codex R1 #9.)_
10. `m4-beforechange` — shared `beforeChange` hook: derive the active ChangeSet from the principal, stamp `changeSetId`; **no active ChangeSet → throw Forbidden**. Guard `!req.user` and hook re-entrancy.
11. `m4-system-deny` — system/Jobs writes deny-by-default. The typed `req.context.systemPurpose` is **narrowly scoped, not a general bypass (Codex R2 #5):** the **bootstrap** purpose may create only the initial tenant / service principal / human login / initial ChangeSet **and** the one named **publish-baseline** operation (the seeded starter page, step 22) — it can **NOT** create or update tenant content generally. All ordinary tenant content (agent or admin) still goes through the ChangeSet write path. Tests prove: (a) a system call without the purpose can't create tenant content; (b) the bootstrap purpose can't be used to write arbitrary tenant content beyond the named baseline op.
12. `m4-ensure-changeset` — the **lower-level transaction contract** the adapter's write primitive calls: within one DB transaction, ensure-or-create the active ChangeSet and stamp it, returning the id the content write uses. Defines the ghost-free semantics (hook can't throw before ensure; a failed write rolls back the ensure). Auto-delete any zero-write ChangeSet as a backstop. The unified primitive that *uses* this lives in the adapter (step 13). _(Codex R1 #2, R2 #2.)_

**Tool broker (Contract A) — the load-bearing isolation control**
13. `m5-localapi-adapter` — the **single** module that holds the `payload` handle; always Local API with `overrideAccess:false` + the Tenant's **service-principal `user`**. No other module may import `payload`/`getPayload`. It exposes the **one unified write primitive `ensureActiveChangeSetThenWrite(...)`** that all content writes (agent or admin) go through; the primitive **takes the per-Site Postgres advisory lock FIRST**, then runs the ensure-changeset transaction contract (step 12) + the write inside that same lock — so the discard path (step 25), which takes the **same** lock, genuinely cannot race a write. A test asserts a write failure rolls back both the ChangeSet and the content. _(Codex R2 #2, #3.)_
14. `m5-lint-deny` — lint rule failing any other `payload.*`/`getPayload` import anywhere.
15. `m5-isolation-tests` — cross-Tenant isolation tests, run locally (vitest/jest). Beyond the basics (Tenant-A principal cannot read/write Tenant-B content; no-active-ChangeSet write rejected), cover the **higher-risk skeleton shortcuts**: (a) **system-purpose denial** — a system call without the typed bootstrap purpose cannot create tenant content; (b) **preview cross-tenant param tampering** — a preview request carrying another tenant's/ChangeSet's IDs is ignored in favor of the session-derived identity; (c) **invalid-intent no-op** — a malformed Gemini intent changes nothing; (d) **discard/write race** — a write in flight during discard cannot recreate draft state. _(Codex R1 #10.)_
16. `m5-allowlist` + **strict intent schema (in slice 1, not deferred)** — per-mode intent allowlist hard-bound to current Tenant + Site + ChangeSet; content mode = CRUD on content collections only, no schema/migration ops. The agent→broker **intent is a strict, closed schema validated at the broker boundary**: parse failure, unknown fields, and out-of-scope targets are **rejected** (fail-closed → the honest "nothing was changed" path), never coerced. This is the load-bearing safety control for a weaker tool-caller (Gemini), so its schema + parse-fail/unknown-field/bad-model-output tests are **slice-1 scope**. _(Codex R1 #7.)_

**Agent (Contract F, Gemini)**
17. `m6-model-seam` — `{provider, modelSlug}` config; default **Gemini Pro**. Claude/OpenRouter drop in later via config only.
18. `m6-orchestrator` — emits intents to the broker, never touches a store directly.
19. `m6-content-agent` — interpret a content request → emit a broker intent; honest failure path ("I couldn't apply that — nothing was changed").
20. `m6-sse` (lite) — stream agent micro-states (Thinking… → Applying… → Updating preview… → Done) to the UI via SSE.

**Auth + workspace UI**
21. `m11-tenant-auth` — Payload user auth, multi-tenant scoped; one login gates the workspace.
22. **Seed script** _(slice-1 substitute for Module 3)_ — insert one Tenant + a **minimal-role service principal** (CRUD on its own content only, NOT admin) + one human Tenant login. Two corrections from review:
    - **Starter content is a REAL published baseline through a real ChangeSet, not a faked status** — to honor `NOT NULL changeSetId` and make "Published" + "discard-back-to-starter" actually true, the seed creates the starter page **inside a ChangeSet** and **publishes it via Payload's supported publish/version path** (a genuine published document/version that preserves `tenant` + `changeSetId`), then marks the ChangeSet terminal. It is NOT a bare `status='published'` flip with no published version behind it. _(Codex R1 #3, R2 #4.)_
    - **Seed uses a narrow, lint-allowlisted bootstrap adapter, not the audited tenant adapter and not a general bypass** — a tiny, separately-reviewed bootstrap surface with a single typed purpose that can ONLY create the initial tenant/principal/login/starter-page and cannot update tenant content generally. It is the one sanctioned second holder of `payload`; the lint-deny (step 14) allowlists exactly this module and nothing else. _(Codex R1 #4.)_
    - No GitHub, no Cloudflare.
23. `m13-workspace` + `m13-mode-toggle` + `m13-state-model` (lite) — chat-left / preview-right shell, explicit "Edit content" / "Change layout" toggle ("Change layout" shows a "coming soon" stub), bottom draft-status bar (`● N draft changes not yet live`), a starter page so the preview is never blank. Visible states limited to **Published** / **Draft pending** (no Publishing/Reviewing — there's no publish yet).
24. **In-Brain preview render** _(slice-1 substitute task `m8-inbrain-preview`)_ — a Brain route renders the active ChangeSet's **draft** content into the right-hand pane; refreshes after each write. **Tenant-leak-safe read (Codex R1 #5):** the preview-read **derives the Tenant from the logged-in human session**, **ignores any client-supplied tenant/ChangeSet IDs**, **pins the active ChangeSet server-side**, reads through the audited adapter (tenant-scoped, never a second DB handle), and returns an **allowlisted public DTO** (no internal/auth/tenant-metadata fields) — the same posture the real preview will need.
25. **Lite discard** _(partial `m13-discard-ux`)_ — discard the active ChangeSet's drafts and return to Published, under a **per-Site Postgres advisory lock shared with the write path** so an in-flight agent/editor write cannot race the discard and recreate draft state (full heartbeat leases stay deferred, but the mutex is slice-1 scope). _(Codex R1 #8.)_ (No branch/preview teardown or media GC — none exist in slice 1.)

**Definition of done (slice 1):** with the seeded Tenant, log in → "change the hero heading to 'Summer Sale'" → see it in preview within seconds → discard → back to starter. All of the above runs on `localhost` against installed Postgres. Green tests, including the **core visible-state-cycle e2e test (Codex R2 #6):** seed creates a **real published** starter → the first edit opens an **active** draft → preview shows the draft → **discard restores the published version** → **no active ChangeSet remains** (and no ghost/zero-write ChangeSet). Plus the isolation/no-ChangeSet-write/system-purpose/preview-tamper/invalid-intent/discard-race tests (steps 11, 15).

## Key decisions & tradeoffs
- **Vertical-slice-first via the existing tracker, not module order and not horizontal build-waves** (Grill Q1) — module order dead-ends on cross-module deps; horizontal waves delay any working result. Trade: the slice cherry-picks tasks, leaving modules partially done (tracked).
- **Native installed Postgres for local dev** (Grill Q2) — user's existing setup. Trade: less disposable than a container; mitigated by the reset-and-reseed script. Prod swap to Neon = connection string only.
- **In-Brain preview now; real Astro-SSR-on-Workers preview + draft-media proxy later** (Grill Q3 → B) — proves the core loop without Cloudflare. Trade: the slice-1 preview UI is throwaway. Deferred set logged in `PENDING.md`.
- **Gemini Pro for the content agent; Claude/structural agent deferred** (Grill Q4) — driven by which key actually exists. The `{provider, modelSlug}` seam makes Claude a one-line add. Trade: Gemini's tool-calling is weaker → mitigated by a strict broker intent schema.
- **Minimal seed script instead of `provisionTenant()`** (Grill Q5 → B) — slice 1 needs one Tenant's data, not an onboarding pipeline. The seed honors the minimal-role service principal so the skeleton doesn't cheat isolation.
- **Local-only; slice stops at preview, no publish** (Grill Q6 → B) — Publish is the whole Module 9 saga + Module 10 media; out of scope here. Trade: no public go-live in slice 1 (by design).
- **Isolation controls are NOT simplified even in the skeleton** — single audited adapter (`overrideAccess:false`), `beforeChange` hook + system-write deny, minimal-role principal, `NOT NULL tenant` FK, one-active-ChangeSet partial unique index, cross-Tenant tests. These are the point of slice 1.

## Risks / open questions
- **Single Local-API adapter is the load-bearing tenant-isolation control.** A bug = cross-Tenant leak. Mitigated by: one exported handle, lint-deny on other imports, mandatory `overrideAccess:false` + service-principal `user`, and CI/local isolation tests. (Inherited from `Architecture.md` Risks.)
- **Blocking-set drift** between the partial unique index and the app guard — single-source the status list as one shared constant + assert it in a test.
- **Gemini structured-output reliability** — a weak tool-call could produce a malformed intent. Mitigated by strict broker-side intent validation + the honest "nothing was changed" failure path. Needs a test with a deliberately bad model output.
- **`beforeChange` hook + first-write atomicity** — ensuring the ChangeSet and the write in one transaction without re-entrancy bugs is fiddly; covered by `m4-ensure-changeset` + a test that a cancelled first edit creates no ghost ChangeSet.
- **Payload v3 on Windows + native Postgres** — version pinning and migration behavior should be verified early (the spike notes open template bugs); slice 1 surfaces this immediately at `m1-scaffold`.
- **System-write deny vs. the seed script** — the seed must use the typed `systemPurpose` allowlist, not a blanket bypass, or it weakens the control it's supposed to respect.
